# 99 AKTAU — руководство по передаче

## Project overview

99 AKTAU — коммерческий сайт PlayStation-клуба на Vite, React и TypeScript.
Фронтенд развёртывается в Vercel, серверные API работают как Vercel Serverless
Functions, а заявки на бронирование хранятся в Supabase. Проект включает гостевую
форму бронирования, отображение статуса заявки, серверную авторизацию
администратора и операции принятия, отклонения, удаления и очистки заявок.

Клиентские страницы не должны получать service role key, пароль администратора
или секрет сессии. Все административные операции выполняются серверными API.

## Environment variables

В проекте используются только следующие серверные переменные:

```env
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=
ADMIN_PASSWORD=
ADMIN_SESSION_SECRET=
```

### `SUPABASE_URL`

- Создаётся Supabase и находится в Project Settings → API.
- Используется серверным Supabase-клиентом для обращения к проекту.
- URL сам по себе не даёт административных прав, но его следует хранить в
  настройках окружения и никогда не объединять в документации с ключами.
- Меняется при переходе на другой Supabase-проект или после его пересоздания.

### `SUPABASE_SERVICE_ROLE_KEY`

- Создаётся Supabase и выдаётся в Project Settings → API как server-side key.
- Используется только Vercel Functions для работы с таблицей и RPC.
- Обходит RLS и поэтому не должен попадать в браузер, Git, документацию, логи или
  переменную с префиксом `VITE_`.
- Обязательно меняется после передачи, утечки или изменения состава участников.

### `ADMIN_PASSWORD`

- Создаётся владельцем и хранится в менеджере паролей.
- Сравнивается только серверным API входа администратора.
- Не должен находиться в клиентском коде, Git, README, issue или обычных чатах.
- Меняется при передаче, подозрении на компрометацию и прекращении доступа
  администратора.

### `ADMIN_SESSION_SECRET`

- Создаётся владельцем как криптографически случайная строка не короче 32 символов.
- Используется сервером для подписи cookie административной сессии.
- Публикация позволяет атаковать механизм сессий; значение хранится только в
  защищённом окружении.
- Меняется после передачи, утечки и при необходимости завершить все старые сессии.

### Generating `ADMIN_SESSION_SECRET`

Node.js:

```bash
node -e "console.log(require('crypto').randomBytes(48).toString('base64'))"
```

OpenSSL:

```bash
openssl rand -base64 48
```

Сгенерированное значение сразу сохраните в менеджере секретов. Не вставляйте его
в Git, документы, issue, pull request или сообщения коммитов.

## Local development

1. Установите Node.js LTS и клонируйте репозиторий.
2. Установите зафиксированные зависимости:

   ```bash
   npm ci
   ```

3. Создайте локальный файл окружения из шаблона и заполните его собственными
   значениями. `.env.local` не должен попадать в Git:

   ```powershell
   Copy-Item .env.example .env.local
   ```

   Для macOS/Linux:

   ```bash
   cp .env.example .env.local
   ```

4. Для работы только с интерфейсом запустите:

   ```bash
   npm run dev
   ```

5. Для локальной проверки Vercel Functions используйте авторизованный Vercel CLI
   и собственное локальное окружение:

   ```bash
   npx vercel dev
   ```

Не используйте production-базу для тестовых заявок без согласования и резервной
копии.

## Validation

Перед каждым релизом и передачей выполните:

```bash
npm run lint
npx tsc --noEmit
npm test
npm run build
npm run handover:check
npm audit
```

Не применяйте `npm audit fix --force`. Уязвимые зависимости обновляйте точечно и
повторяйте весь набор проверок.

## GitHub setup and handover

1. Убедитесь, что репозиторий относится к 99 AKTAU и ветка `main` защищена.
2. Проверьте отсутствие `.env`, резервных копий, баз данных, build-артефактов и
   личных файлов в Git.
3. Создайте резервную копию репозитория и зафиксируйте hash принятого релиза.
4. Добавьте GitHub-аккаунт клиента или выполните передачу владельцем вручную.
5. Включите 2FA, ограничьте права участников и настройте branch protection.
6. Подключайте Vercel только к проверенной ветке `main` этого репозитория.
7. После подтверждения клиента удалите старые доступы и отзовите старые токены.

Передача владельца репозитория, удаление участников и изменение GitHub Secrets
выполняются владельцем вручную и не являются частью автоматической проверки.

## Production deployment in Vercel

1. Создайте Vercel-проект в аккаунте клиента и импортируйте GitHub-репозиторий.
2. Выберите production-ветку `main`.
3. Укажите build command `npm run build` и output directory `dist`.
4. Добавьте четыре переменные окружения из раздела выше отдельно для Production.
   Не копируйте их в документацию и не делайте клиентскими `VITE_` переменными.
5. Запустите deployment и проверьте build/runtime logs без публикации секретов.
6. Проверьте страницы `/`, `/about`, `/zones`, `/booking`, `/contacts`, а также
   гостевое бронирование, вход администратора и изменение статуса заявки.
7. Зафиксируйте deployment URL и commit hash в закрытом акте приёмки.

Не меняйте production-переменные, домен или настройки доступа без резервной копии
и согласования с владельцем.

## Supabase setup

1. Создайте Supabase-проект в аккаунте клиента и сохраните recovery-информацию.
2. В SQL Editor выполните [supabase/schema.sql](supabase/schema.sql).
3. Убедитесь, что созданы таблица `public.bookings`, индексы и функция
   `public.create_booking_request`.
4. Проверьте, что RLS включён, публичные политики удалены, а права на таблицу и
   RPC выданы только роли `service_role`.
5. В Project Settings → API получите project URL и server-side service role key.
6. Добавьте значения в защищённые переменные Vercel; не вставляйте service role
   key во frontend, `VITE_` переменные или ответы API.
7. Выполните тестовое бронирование и административные действия на согласованной
   среде, затем удалите тестовые данные штатным административным действием.

Если используется существующий production-проект, до любых SQL-изменений
создайте проверяемый backup. Не удаляйте и не изменяйте production-данные в ходе
передачи без отдельного согласования.

## Admin guide

1. Откройте сайт и нажмите «Вход для администратора» в меню.
2. Введите `ADMIN_PASSWORD` из защищённого менеджера паролей.
3. После входа откройте `/booking`: справа появится список заявок.
4. «Принять» переводит заявку в статус accepted.
5. «Отклонить» переводит заявку в статус rejected.
6. «Удалить» удаляет выбранную заявку.
7. «Очистить список заявок» удаляет все заявки после подтверждения.
8. По окончании работы выполните выход администратора и проверьте, что список и
   административные действия больше недоступны.

Пароль администратора никогда не добавляется в это руководство.

## Domain setup

1. Домен должен находиться в аккаунте клиента у регистратора.
2. В Vercel откройте Project Settings → Domains и добавьте домен.
3. Внесите только DNS-записи, которые показывает Vercel, в аккаунте регистратора.
4. Дождитесь подтверждения домена и выпуска HTTPS-сертификата.
5. Выберите основной домен, проверьте редиректы, canonical URL, `robots.txt` и
   `sitemap.xml`.
6. Обновите Google Search Console и аналитику для production-домена.

DNS изменяет только владелец домена. Сохраните прежние записи перед изменениями.

## Backup and recovery

### Код

- Убедитесь, что `main` отправлена в GitHub и сохраните commit hash релиза.
- Создайте отдельную защищённую копию репозитория или release-архив без `.env`.
- Периодически проверяйте возможность клонирования и production build из копии.

### База данных

- Перед передачей и SQL-изменениями создайте backup средствами Supabase или
  `pg_dump` с учётными данными владельца.
- Храните экспорт в зашифрованном хранилище с ограниченным доступом.
- Сохраните `supabase/schema.sql` вместе с кодом, но не добавляйте дампы данных в Git.
- Проверьте, что backup можно восстановить в отдельный тестовый проект.

### Recovery

1. Восстановите код из проверенного commit и выполните `npm ci` и `npm run build`.
2. Создайте или восстановите Supabase-проект и примените схему/backup.
3. Создайте новые секреты и добавьте их в Vercel.
4. Выполните deployment, функциональную проверку и только затем переключайте домен.

## Security rules

- Никогда не коммитьте `.env`, `.env.local`, `.env.production`, дампы или токены.
- Не публикуйте `SUPABASE_SERVICE_ROLE_KEY`, пароль или session secret.
- Не отправляйте секреты в обычные чаты, документацию, issue или pull request.
- Меняйте секреты после смены владельца и отзывайте старые значения.
- Удаляйте доступ бывших участников только после завершения финальной проверки.
- Используйте уникальный сложный пароль администратора и менеджер паролей.
- Включите 2FA для GitHub, Vercel, Supabase, регистратора и Google.
- Регулярно выполняйте проверки из раздела Validation и просматривайте audit logs.
